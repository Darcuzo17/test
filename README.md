# Твоя миссия

Ты — научный сотрудник на борту научно-исследовательского корабля, потерпевшего крушение на неизвестной планете. Пока техники чинят двигатели, от тебя зависит спасение миссии.
Твоя задача — разработать единое веб-приложение, которое решит три ключевые научные задачи, стоящие перед экипажем.

## Требования к решению
Тебе нужно создать веб-приложение, которое:
- запускает HTTP-сервер на порту 8000;
- предоставляет три API-эндпоинта (по одному для каждой задачи);
- принимает входные данные в формате JSON через POST-запросы;
- выполняет расчеты с учетом всех условий и ограничений;
- возвращает результаты в строго определенном JSON-формате.


Теперь перейдем к конкретным задачам. Каждая из них представляет отдельную научную проблему, которую необходимо решить для спасения экипажа.


# Задача 1: «Круиз Робинсона»
## Легенда
Корабль повреждён, но у нас есть аварийный шаттл и отчаянный агент Робинсон, готовый лететь за помощью. Проблема в том, что взлет с этой планеты требует чудовищных затрат топлива — настолько огромных, что каждый лишний килограмм на орбите стоит целого бака горючего.
## Что мы знаем:
- на вывод единицы массы на орбиту нужно огромное количество топлива (считаем, что человек и баки с запасом кислорода массы не имеют);
- после выхода на орбиту шаттл летит по автопрограмме:
  - половина имеющегося на борту топлива тратится на разгон (с округлением до целых вниз);
    - при этом на разгон допустимо потратить не более половины от имеющегося запаса кислорода. Как только достигнут этот предел - ускорение прекращается
  - затем шаттл летит по инерции (равномерное движение без ускорений);
  - оставшаяся часть топлива тратится на торможение и посадку;
    - время полета по инерции рассчитывается таким образом, чтобы в момент полной остановки шаттла запас кислорода был полностью израсходован;
  - работа двигателей на разгон и торможение осуществляется таким образом, что каждую секунду сжигается одна единица топлива:
    - учитывайте, что чем меньше на борту топлива, тем меньше суммарная масса шаттла, следовательно, тем большее ускорение получит шаттл;
    - считайте, расход единицы топлива происходит мгновенно, после чего на одну секунду шаттл получает соответствующее его текущей (без учета только что потраченного топлива) массе ускорение;
    - на последней секунде торможения двигатели могут работать меньше секунды, чтобы не придать шаттлу скорость в обратном направлении; при этом все равно тратится полная единица топлива и ровно секунда на выполнение торможения (оставшееся время шаттл неподвижен)
- на борту ограничен запас воздуха — полет не может длиться вечно.


## Местные жители и «Звездный Ключ»


Впрочем, местные жители готовы помочь нам сэкономить часть топлива на взлете и предлагают воспользоваться их технологией запуска — «Звездный Ключ», с помощью которой можно придать шаттлу начальный импульс, и чем он будет сильнее, тем ниже окажутся затраты топлива на взлете; как выражаются местные – импульс способен вывести часть массы на орбиту, остальное нам придется доставить самим. И чтобы зарядить «Звездный Ключ», они хотят получить наше топливо.


В репозитории в папке ```/materials``` находится готовый алгоритмический модуль ```star_key.rs```. Он рассчитывает, какую массу местные жители согласны вывести на орбиту в обмен на переданное топливо.


Модуль **нельзя изменять**! Твоя задача — изучить предоставленный  модуль, интегрировать его в модель полета (подключить Rust-модуль в решение и обращаться к нему, как к библиотеке) и найти оптимальный план: сколько топлива выгоднее потратить самим, а сколько — обменять у местных, чтобы улететь максимально далеко.


## Что нужно сделать в этой задаче
Написать код, который:
- интегрирует предоставленный модуль ```/materials/star_key.rs``` в модель полета;
- найдет оптимальный баланс между самостоятельным расходом топлива на взлет и обменом с местными жителями;
- рассчитает максимальное расстояние полета шаттла с учетом всех ограничений.


Тебе нужно найти баланс между:
- массой топлива на борту (чем больше — тем дальше полетим);
- ограниченным запасом воздуха (полет не может превышать заданное время);
- огромными затратами на вывод на орбиту (чем больше масса — тем больше топлива сжигается при взлете).


## API-интерфейс задачи


Все три задания в рамках Проектного этапа должны составлять **ОДНО** веб-приложение. Приложение должно запускать HTTP-сервер на порту 8000 и обрабатывать запрос (для задачи 1): ```POST /api/v1/robinson_cruise```


Формат запроса (JSON):
```json
{
  "mass_shuttle": 1000.0,
  "mass_fuel_unit": 1.0,
  "power_per_unit": 100.0,
  "oxygen_time": 8640,
  "total_fuel": 4000,
  "fuel_consumption": 3.0
}
```
Формат ответа (JSON):
```json
{
  "max_distance": 107001.1
}
```
Код ответа: ```200 OK```

Формат ответа при вводе некорректного формата или диапазона значений (JSON):
```json
{
  "status": "incorrect_input"
}
```
Код ответа: ```400 Bad Request```

Где:
- ```mass_shuttle``` — масса шаттла без топлива (кг);
- ```mass_fuel_unit``` — масса единицы топлива (кг);
- ```power_per_unit``` — ускорение, прилагаемое к единице массы путем расхода одной единицы топлива (кг*м/с²);
- ```oxygen_time``` — время, на которое хватит воздуха (секунды, целое число);
- ```total_fuel``` — общий запас топлива, который можно потратить на весь маневр (целые ед.);
- ```fuel_consumption``` — расход топлива – сколько единиц топлива сжигается за самостоятельный (без помощи местных) вывод единицы массы на орбиту;
- ```max_distance``` — максимальное расстояние полета (метры, одна цифра после запятой); гарантируется, что это расстояние не превышает _1e9_ метров;
- ```status``` — статус ответа приложения.

Учитывай, что важной частью решения задачи является проверка валидности входных данных. В числе возможных ошибок может быть как несоблюдение описанного выше формата (например, отсутствие или неправильное наименование части полей, ошибки в типах данных), так и некорректность значений (например, выход за допустимые диапазоны, нарушение связности данных или несоответствие условиям задачи).  
Автоматическая проверка включает не только тесты на корректную обработку валидных запросов, но и тесты на некорректные входные данные.  
Решение должно корректно обрабатывать такие случаи и возвращать ответ строго в указанном формате.

За решение этой задачи можно получить максимум **25** баллов.


# Задача 2: «Гори, гори, моя звезда»
## Легенда
Корабль получает загадочные сигналы из космоса — возможно, это послание от другой цивилизации или данные о безопасном маршруте. Но постоянно наблюдать за звездой-источником невозможно: несколько лун планеты регулярно закрывают ее от обзора.


Твоя задача — составить точное расписание, когда звезда будет видна, чтобы мы могли настроить аппаратуру на прием и расшифровку сигналов в нужные периоды времени.


## Что мы знаем:
- мы находимся в фиксированной точке наблюдения (точка 0,0 на плоскости);
- интересующая нас звезда находится очень далеко — считаем ее материальной точкой;
- вокруг планеты, на которой мы находимся, (вокруг 0,0) вращается несколько лун — каждая движется по круговой орбите;
- луны — это идеальные сферы определенного радиуса, которые могут полностью или частично заслонять звезду;
- лун не меньше двух;
- луны гарантированно движутся с разными угловыми скоростями, но в одном направлении (по часовой стрелке).


## Сломанная аппаратура и устройство местных жителей
Но не все так просто… Наша измерительная техника была повреждена при крушении, поэтому скорости вращения лун неизвестны — в собранных данных есть только их размеры и позиции на момент начала наблюдения.


К счастью, местные жители предоставили нам свое устройство — "Световой анализатор". Оно позволяет определить для любой пары лун:
- как часто одна луна заслоняет другую (время между концом предыдущего затмения и началом следующего);
- угловая скорость какой из лун больше;
- средний период обращения выбранной пары лун вокруг планеты.


Но есть нюанс: имена лун в устройстве зашифрованы хеш-кодами, которые нужно расшифровать.


В репозитории в папке ```/materials``` находится готовый алгоритмический модуль ```light_analyzer.rs```, который:
- принимает на вход хеши двух лун;
- рассчитывает и возвращает описанный выше набор данных.


Модуль **нельзя изменять**, только интегрировать в решение!


## Что нужно сделать в этой задаче
Написать код, который:
- восстановит угловые скорости всех лун, используя данные от устройства местных жителей о взаимных затмениях;
- рассчитает периоды, когда вектор от точки  0.0 на звезду пересекается со сферами лун;
- составит для указанной даты расписание интервалов времени, когда звезда видна.


## API-интерфейс задачи
Напоминаем, что все три задания в рамках Проектного этапа должны составлять **ОДНО** веб-приложение. Приложение должно запускать HTTP-сервер на порту 8000 и обрабатывать запрос (для задачи 2): ```POST /api/v1/star_visibility```
Формат запроса (JSON):
```json
{
  "star_vector": {"x": 0.0, "y": 1.0},
  "moons": [
    {
      "name": "Moon1",
      "orbit_radius": 3.4e8,
      "moon_radius": 1.7e6,
      "initial_observation_angle": 60.0,
      "initial_observation_time": "2026-03-14T00:00:00Z"
    },
    {
      "name": "Moon2",
      "orbit_radius": 8.1e8,
      "moon_radius": 1.1e6,
      "initial_observation_angle": 144.2,
      "initial_observation_time": "2026-03-14T00:00:00Z"
    }
  ],
  "observation_date": "2026-03-15"
}
```
Формат ответа (JSON):
```json
{
  "visible_intervals": [
    {
      "start": "2026-03-15T08:30:00Z",
      "end": "2026-03-15T10:00:00Z"
    },
    {
      "start": "2026-03-15T15:40:01Z",
      "end": "2026-03-15T15:40:02Z"
    }
  ]
}
```
Код ответа: ```200 OK```

Формат ответа при вводе некорректного формата или диапазона значений (JSON):
```json
{
  "status": "incorrect_input"
}
```
Код ответа: ```400 Bad Request```

Где:
- ```star_vector``` — координаты вектора, задающего направление на звезду, за которой мы наблюдаем;
- ```moons``` — данные о лунах (имя, радиус орбиты, радиус луны, начальная позиция и время), _2 <= len(moons) <= 100_;
- ```name``` — название луны;
- ```orbit_radius``` — радиус вращения центра луны;
- ```moon_radius``` — радиус луны;
- ```initial_observation_angle``` — начальное положение центра луны на орбите в момент времени `initial_observation_time` (в градусах, относительно положительной оси X в направлении против часовой стрелки, `0 <= initial_observation_angle < 360`);
- ```initial_observation_time``` — начальное время (время, когда мы определили начальную позицию луны);
- ```visible_intervals``` — интервалы времени, когда наблюдаемая звезда не закрыта ни одной из лун; интервалы должны быть упорядочены по времени начала интервала;
- ```observation_date``` - дата, для которой необходимо рассчитать интервалы видимости звезды;
- ```status``` — статус ответа приложения;

Обрати внимание, что в качестве выходных данных необходимо указать интервалы, округленные до целых секунд; при этом считается, что на всем указанном временном отрезке (включая правую границу) звезда постоянно видна. Например, если никаких затмений в указанный день нет, то ответом будет один интервал ровно в 24 часа, от `00:00:00` и по `23:59:59`.

Учитывай, что важной частью решения задачи является проверка валидности входных данных. # Задача 3: «Звездочетность»
## Легенда
Корабль потерял ориентацию в пространстве. Чтобы определить свое местоположение в галактике и построить обратный курс, нужно найти на небе знакомое созвездие — тот самый астронавигационный маркер, по которому можно сориентироваться.


Проблема в том, что мы не знаем, в какой части галактики находимся, и поэтому созвездие, которое мы ищем, может быть произвольным образом искажено в пространстве, и поэтому мы не можем полагаться на визуальное сходство, а должны найти похожую по структуре конфигурацию звезд.

Под похожей конфигурацией пары созвездий мы будем понимать совпадение сразу двух свойств графов, которые задают созвездия:
- графы должны быть изоморфны друг другу;
- мы должны определить взаимно-однозначное соответствие между вершинами (звездами) первого и второго созвездий, и для этого потребуется проанализировать длины ребер: они могут отличаться в силу пространственных искажений, однако относительный порядок сохраняется. 

Например, мы наблюдаем структуру `a - (10) - b - (20) - c` на первом графе и `x - (5) - y - (50) - z` на втором (в скобках обозначены длины ребер). Мы понимаем, что эти два графа изоморфны друг другу, а по относительной величине ребер (ребро `xy` меньше ребра `yz`, как и ребро `ab` меньше, чем `bc`) можно сделать вывод, что мы можем сопоставить звезды так: `a->x`, `b->y`, `c->z`.

Более того — наша измерительная аппаратура повреждена. Мы видим звезды, но не можем измерить расстояния между ними напрямую.


## Что мы знаем:
- на небе видно множество звезд, каждая имеет уникальное местное имя (в виде хеш-кода);
- искомое созвездие описано как неориентированный граф: какие звезды соединены между собой;
- понять, является ли созвездие на небе тем, которое нам требуется, можно только определив изоморфность их графов и сопоставив длины ребер;
- координаты звезд неизвестны — только их хеши.


## Устройство местных жителей
Местные жители предоставили нам устройство — "Галактический идентификатор". С его помощью можно получить информацию о любых двух звездах:
- находятся ли они в одном созвездии;
- расстояние между ними (если они в одном созвездии).

Следует учитывать, что на небе могут быть изолированные звезды, не включенные ни в одно созвездие.

Похоже, что жители этой планеты используют те же принципы построения схем созвездий, что и мы - соединяют звезды в связный рисунок таким образом, чтобы сумма длин ребер оказалась минимальной и не содержала циклов.


В репозитории в папке ```/materials``` находится готовый алгоритмический модуль  ```galactic_identifier.rs```. Модуль:
- принимает на вход хеши двух звезд;
- возвращает информацию о том, находятся ли они в одном созвездии, и если да — расстояние между ними.


Модуль **нельзя изменять**, только интегрировать в твое решение!


## Что нужно сделать в этой задаче
Написать код, который:
- используя попарные запросы к “Галактическому идентификатору”, кластеризует звезды по созвездиям;
- восстановит матрицу расстояний внутри каждого созвездия;
- восстановит схему (граф) наблюдаемых созвездий;
- найдет на небе созвездие, схема которого была бы изоморфна схеме заданного созвездия.


Особые условия:
- созвездия представляют собой минимальные остовные деревья (MST);
- гарантируется, что для каждого созвездия существует единственный способ построения MST.


## API-интерфейс задачи
Напоминаем, что все три задания в рамках Проектного этапа должны составлять **ОДНО** веб-приложение. Приложение должно запускать HTTP-сервер на порту 8000 и обрабатывать запрос (для задачи 3): ```POST /api/v1/constellation_finder```


Формат запроса (JSON):
```json
{
  "star_hashes": ["hash1", "hash2", "hash3", ...],
  "target_constellation": {
    "edges": [
      {"from": 0, "to": 1, "distance": 1e9},
      {"from": 1, "to": 2, "distance": 2e9},
      {"from": 2, "to": 3, "distance": 3e9}
    ]
  }
}
```
Формат ответа (JSON):
```json
{
  "found": true,
  "matched_stars": ["hash7", "hash23", "hash15"]
}
```
Код ответа: ```200 OK```

Или если не найдено:
```json
{
  "found": false
}
```
Код ответа: ```200 OK```

Формат ответа при вводе некорректного формата или диапазона значений (JSON):
```json
{
  "status": "incorrect_input"
}
```
Код ответа: ```400 Bad Request```

Где:
- ```star_hashes``` — массив хешей всех видимых звезд, _0 <= len(star_hashes) <= 1000_;
- ```target_constellation``` — описание искомого созвездия (граф из n вершин с индексами [0…n-1], _2 <= n <= 50_); описание представляет собой список ребер, каждое из которых задается индексами соединенных вершин и длиной ребра;
- ```found``` — найдено ли созвездие. Если созвездий, изоморфных искомому, оказалось несколько – считайте, что найти не удалось;
- ```matched_stars``` — хеши звезд, соответствующих вершинам целевого созвездия (в порядке от 0 до n-1);
- ```status``` — статус ответа приложения.

Учитывай, что важной частью решения задачи является проверка валидности входных данных. В числе возможных ошибок может быть как несоблюдение описанного выше формата (например, отсутствие или неправильное наименование части полей, ошибки в типах данных), так и некорректность значений (например, выход за допустимые диапазоны, нарушение связности данных или несоответствие условиям задачи).  
Автоматическая проверка включает не только тесты на корректную обработку валидных запросов, но и тесты на некорректные входные данные.  
Решение должно корректно обрабатывать такие случаи и возвращать ответ строго в указанном формате.

Примечание: использованные в описании формата запроса имена звезд **НЕ** являются валидными примерами хешей, которые обрабатывает Галактический Идентификатор.
